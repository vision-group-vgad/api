2 · Prerequisites

  • A DigitalOcean account — cloud.digitalocean.com
  • A domain name (e.g., visiongroup.co.ug) with DNS you can manage
  • Your backend and frontend code pushed to GitHub (repos: vision-group-vgad/api and vision-group-vgad/vgad)
  • SSH key pair — generate on Windows with: ssh-keygen -t ed25519 -C "deploy-key"
  • All .env values saved somewhere safe (see Section 9)

Steps in the DigitalOcean Dashboard

  1. Go to Create → Droplet
  2. Choose Ubuntu 24.04 LTS x64
  3. Select your droplet size (table above)
  4. Choose the region closest to your users (e.g., London or Frankfurt for East Africa)
  5. Under Authentication → paste your SSH public key
  6. Set a hostname: e.g., vgad-api or vgad-app
  7. Click Create Droplet — note the public IP address shown

Configure DNS

In your domain registrar (or DigitalOcean DNS), create A records:

Record TypeNameValue
Aapi.yourdomain.comBackend droplet IP
Aapp.yourdomain.com or @Frontend droplet IP (or same IP for Option B)
Note: DNS propagation can take up to 48 hours, but usually works within 5–15 minutes with DigitalOcean DNS.

4 · Server Setup (Run on Every Droplet)

Connect via SSH: ssh root@YOUR_DROPLET_IP

4.1 — Update the system

apt update && apt upgrade -y

4.2 — Install Node.js 20

curl -fsSL https://deb.nodesource.com/setup_20.x | bash -
apt install -y nodejs
node -v    # Should show v20.x.x
npm -v

4.3 — Install Git, Nginx, and PM2

apt install -y git nginx
npm install -g pm2
pm2 startup systemd   # Copy and run the output command it gives you

4.4 — Create an application user (recommended — do not run as root)

adduser --disabled-password --gecos "" vgad
usermod -aG sudo vgad
mkdir -p /var/www
chown vgad:vgad /var/www

5 · Deploy the Backend API

Run these on the backend droplet (or the single droplet for Option B).

5.1 — Clone the repository

su - vgad
cd /var/www
git clone https://github.com/vision-group-vgad/api.git
cd api

5.2 — Install dependencies

npm install --omit=dev

5.3 — Create the .env file

Important: Never commit your .env file to GitHub. Create it manually on the server.
nano /var/www/api/.env

Paste in all your environment variables (see Section 9 for the full list). Minimum required to start:

PORT=4000
SECRET_KEY=your_jwt_secret_key_here
DATABASE_URL=postgresql://user:pass@host:5432/dbname
CMS_API_KEY=your_cms_api_key
CMC_API_BASE_URL=https://cms-vgad.visiongroup.co.ug/api
CMC_API_BEARER_TOKEN=your_bearer_token
ALLOWED_ORIGIN=https://app.yourdomain.com
ALLOWED_ORIGINS=https://app.yourdomain.com

5.4 — Start with PM2

cd /var/www/api
pm2 start server.js --name vgad-api --watch false
pm2 save
pm2 status

Verify the API is running internally:

curl http://localhost:4000/api/v1/auth/login -X POST \
  -H "Content-Type: application/json" \
  -d '{"email":"headofsales@vision.com","password":"SalesHead@2024!"}'
# Should return a JSON token

5.5 — Updating the backend (re-deploying)

cd /var/www/api
git pull origin main
npm install --omit=dev
pm2 restart vgad-api

6 · Deploy the Frontend

Run these on the frontend droplet (or the same single droplet for Option B).

6.1 — Clone the repository

su - vgad
cd /var/www
git clone https://github.com/vision-group-vgad/vgad.git
cd vgad

6.2 — Create the .env file for the build

Important: Vite bakes environment variables into the built static files at build time. You must set these before running npm run build.
nano /var/www/vgad/.env

Contents — replace with your real domain:

# Base URL used by globalApi.tsx (axios instance)
VITE_API_BASE_URL=https://api.yourdomain.com

# Base URL used by salesApi.tsx, executiveApi.tsx, finianceAPis.tsx
VITE_API_BASE=https://api.yourdomain.com/api/v1
Single Droplet (Option B): If both apps are on the same server, you can use the same domain with a path prefix: VITE_API_BASE_URL=https://yourdomain.com and VITE_API_BASE=https://yourdomain.com/api/v1

6.3 — Install dependencies and build

cd /var/www/vgad
npm install
npm run build

This produces a /var/www/vgad/dist/ folder — these are the static files Nginx will serve.

6.4 — Verify the build output

ls -la /var/www/vgad/dist/
# Should show: index.html, assets/, etc.

6.5 — Updating the frontend (re-deploying)

cd /var/www/vgad
git pull origin main
npm install
npm run build
# Nginx picks up the new files automatically — no restart needed.

7 · Configure Nginx Reverse Proxy

Option A — Two Droplets (Backend Nginx)

Create /etc/nginx/sites-available/vgad-api:

server {
    listen 80;
    server_name api.yourdomain.com;

    location / {
        proxy_pass         http://localhost:4000;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection 'upgrade';
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;

        # WebSocket support (Socket.IO notifications)
        proxy_read_timeout 86400;
    }
}

Create /etc/nginx/sites-available/vgad-frontend on the frontend droplet:

server {
    listen 80;
    server_name app.yourdomain.com;
    root /var/www/vgad/dist;
    index index.html;

    # React Router — send all routes to index.html
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache static assets for 1 year
    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    gzip on;
    gzip_types text/plain text/css application/json application/javascript;
}

Option B — Single Droplet (one Nginx config)

Create /etc/nginx/sites-available/vgad:

server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;
    root /var/www/vgad/dist;
    index index.html;

    # API — proxy to Node.js backend
    location /api/ {
        proxy_pass         http://localhost:4000;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection 'upgrade';
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
        proxy_read_timeout 86400;
    }

    # Socket.IO WebSocket path
    location /socket.io/ {
        proxy_pass         http://localhost:4000;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection 'upgrade';
    }

    # Frontend — serve static files, React Router fallback
    location / {
        try_files $uri $uri/ /index.html;
    }

    location /assets/ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    gzip on;
    gzip_types text/plain text/css application/json application/javascript;
}

Enable the site and test

# Link site config and disable default
ln -s /etc/nginx/sites-available/vgad /etc/nginx/sites-enabled/
rm -f /etc/nginx/sites-enabled/default

# Test configuration
nginx -t

# Restart Nginx
systemctl restart nginx
systemctl enable nginx

8 · Enable HTTPS with Let's Encrypt

Why HTTPS? Browsers block mixed-content (HTTP API calls from an HTTPS page). You need SSL on both frontend and API.

8.1 — Install Certbot

apt install -y certbot python3-certbot-nginx

8.2 — Obtain SSL certificates

Two droplets: Run on each droplet with its own domain:

certbot --nginx -d api.yourdomain.com
certbot --nginx -d app.yourdomain.com

Single droplet:

certbot --nginx -d yourdomain.com -d www.yourdomain.com

Follow the prompts — Certbot will automatically update your Nginx config to redirect HTTP → HTTPS.

8.3 — Auto-renewal

systemctl enable certbot.timer
# Test renewal:
certbot renew --dry-run
After getting SSL: Update your .env on the frontend build machine to use https:// URLs, then rebuild and redeploy. Also update ALLOWED_ORIGIN and ALLOWED_ORIGINS in the backend .env to use https://.

10 · PM2 Process Management

Useful PM2 commands

CommandWhat it does
pm2 statusShow all running processes and their status
pm2 logs vgad-apiStream live logs for the API
pm2 logs vgad-api --lines 200Show last 200 log lines
pm2 restart vgad-apiRestart the API process
pm2 stop vgad-apiStop the API process

pm2 monitLive dashboard of CPU/RAM/logs

Create a PM2 ecosystem file (easier config)

Create /var/www/api/ecosystem.config.cjs:

module.exports = {
  apps: [{
    name: "vgad-api",
    script: "server.js",
    cwd: "/var/www/api",
    env_production: {
      NODE_ENV: "production",
      PORT: 4000
    },
    instances: 1,
    exec_mode: "fork",
    watch: false,
    max_memory_restart: "400M",
    error_file: "/var/log/pm2/vgad-api-error.log",
    out_file: "/var/log/pm2/vgad-api-out.log",
    merge_logs: true,
    log_date_format: "YYYY-MM-DD HH:mm:ss"
  }]
};

pm2 start ecosystem.config.cjs --env production
pm2 save

11 · Firewall Rules

Use UFW (Uncomplicated Firewall)

ufw allow OpenSSH
ufw allow 'Nginx Full'    # Allows ports 80 and 443
ufw enable
ufw status
Port 4000 should NOT be publicly exposed. Nginx acts as a reverse proxy so all traffic goes through port 80/443. Port 4000 should only be accessible internally (from Nginx on the same server, or between private network droplets). Do NOT run ufw allow 4000.

DigitalOcean Firewall (additional layer)

In the DigitalOcean dashboard, go to Networking → Firewalls and create a cloud firewall:

Inbound RuleProtocolPortSource
HTTPTCP80All IPv4, All IPv6
HTTPSTCP443All IPv4, All IPv6
SSHTCP22Your office IP only

12 · Troubleshooting

ProblemWhat to check
API returns 502 Bad Gateway Node.js may not be running. Check with:
pm2 status
pm2 logs vgad-api --lines 50
Then restart: pm2 restart vgad-api
Frontend shows blank page Check the browser console. Usually a wrong VITE_API_BASE_URL. Rebuild the frontend with the correct .env values.
CORS errors in browser console Backend ALLOWED_ORIGINS must exactly match the frontend URL (including https:// and no trailing slash). Restart the API after changing .env.
Login works but dashboard shows no data The CMS API may be unreachable from the server. Test: curl -H "Authorization: Bearer YOUR_CMS_KEY" https://cms-vgad.visiongroup.co.ug/api/bc-datasets
Nginx test fails (nginx -t) Check for syntax errors in your site config file. Logs: cat /var/log/nginx/error.log | tail -30
SSL certificate fails (Certbot) Ensure DNS A record points to the droplet IP. Check with: dig +short api.yourdomain.com. Wait for propagation if recently changed.
After reboot, API not running Run pm2 save and pm2 startup systemd, then execute the command it outputs. This makes PM2 auto-start on boot.
Port 4000 shows "connection refused" This is expected if you query it from outside. Nginx should be the only public entry point. From inside the server: curl http://localhost:4000/api/v1/health
Database connection errors Check DATABASE_URL in .env. If using a managed DigitalOcean DB, whitelist the droplet's IP in the DB trusted sources settings.
